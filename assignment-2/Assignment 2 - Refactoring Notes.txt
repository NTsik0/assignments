# Assignment 2 : Refactoring Notes  - Nikoloz Tsikaridze

Each section names the file, states the original smell, lists the refactorings applied, and explains why the result is better.

N:1. Alternative Classes with Different Interfaces
File: AlternativeClassesWithDifferentInterfacesExample.java

Original smell:
ZoomClassroom.beginSession() and TeamsClassroom.openMeeting() served the same purpose but exposed different method names. Client code had to know both APIs and branch between them. Substitution was impossible without adaptation.

Refactorings applied:
 Extract Interface (VirtualClassroom) with a single agreed:upon method start().
 Change Method Signature : both implementations renamed their method to start().
 Use Supertype Where Possible : client code now declares VirtualClassroom references.

Why it is better:
The client is decoupled from concrete providers. Adding a third platform like GoogleMeetClassroom requires only a new VirtualClassroom implementation with no changes to existing client code.

N:2. Comments
File: CommentsExample.java

Original smell:
Three block comments were compensating for a method that mixed three distinct concerns (VIP discount, bulk discount, tax) without naming them. Removing the comments would have left the reader guessing.

Refactorings applied:
 Extract Method : applyVipDiscount(), applyBulkDiscount(), applyTax().
 Rename : the top-level method renamed to priceWithDiscountsAndTax().

Why it is better:
The extracted method names replace the comments entirely. The body of the top-level method is now a four-line narrative that any reader can follow without annotations.

N:3. Data Class
File: DataClassExample.java

Original smell:
StudentRecord was a passive bag of public fields. Three separate helper classes (HonorsEvaluator, TuitionDiscountCalculator, AcademicStandingReporter) held all the logic, leaving the domain model anemic and poorly cohesive.

Refactorings applied:
 Encapsulate Fields : fields made private with a constructor.
 Move Method : isEligibleForHonors(), tuitionDiscountPercent(), and academicStanding() moved into StudentRecord.
 The three helper classes removed; their only job was operating on the record's own data.

Why it is better:
StudentRecord now owns behavior that depends on its own state. The client asks the student object directly. No external class can corrupt the fields. The model is active, not passive.

N:5. Divergent Change
File: DivergentChangeExample.java

Original smell:
One class held report formatting, SQL generation, and CSV export : three completely unrelated reasons to change. A DB schema rename, a new report template, and a spreadsheet tool change would all hit the same class.

Refactorings applied:
 Extract Class : ExamReportFormatter, ExamSqlBuilder, ExamCsvExporter.

N:17. Mutable Data
File: MutableDataExample.java

Original smell:
getEnrolledStudents() returned the live internal list. Any caller could clear(), remove(), or add() entries, silently corrupting the object's state : as clientCode() demonstrated.

Refactorings applied:
 Encapsulate Field : list is private and final.
 Return Unmodifiable View : getEnrolledStudents() wraps the list with Collections.unmodifiableList().
 Encapsulate Updates : enroll() is the only mutation path.

Why it is better:
External code cannot corrupt the enrollment list. All mutations go through enroll(), the natural place for future validation or events. An attempted clear() now throws UnsupportedOperationException, making the contract explicit.

N:20. Refused Bequest
File: RefusedBequestExample.java

Original smell:
Penguin extended Bird and threw UnsupportedOperationException from fly(). Any code holding a Bird reference and calling fly() could blow up at runtime : a direct LSP violation. The hierarchy was a lie.

Refactorings applied:
 Extract Superclass : Bird becomes an abstract base with move() (a locomotion contract all birds honour).
 Push Down : fly() pushed down into a new FlyingBird intermediate class.
 Penguin now extends Bird directly and implements move() as waddling.
 Use Supertype Where Possible : client uses Bird for general locomotion, FlyingBird only when flight is specifically expected.

Why it is better:
No bird throws at runtime. Penguin is a legitimate Bird. FlyingBird genuinely supports flying. The hierarchy matches the real:world relationship and LSP is satisfied.

N:21. Repeated Switches
File: RepeatedSwitchesExample.java

Original smell:
Two switch statements both branched on studentType. Adding a new type required coordinated edits in two separate methods, with a risk of forgetting one.

Refactorings applied:
 Replace Conditional with Polymorphism : StudentType becomes an enum that carries both tuitionDiscount and dormPriority as data for each variant.
 The two switch methods are replaced by direct data lookups on the enum constant.

Why it is better:
Adding a new student type is one enum constant addition; all consuming methods handle it automatically. The duplication is eliminated. New variants extend the enum without modifying any existing switch arms (Open/Closed Principle).

N:24. Temporary Field
File: TemporaryFieldExample.java

Original smell:
examRoom and onlineMeetingLink were both fields on one class, but only one was ever meaningful at a time. Depending on which method was called, part of the object's state was always null.

Refactorings applied:
 Extract Class : OnsiteExam owns only room; OnlineExam owns only meetingLink.
 Move Method : preparation logic moves into the appropriate class.
 The original class becomes a thin coordinator.

Why it is better:
Each class is always in a fully consistent state. OnsiteExam always has a room; OnlineExam always has a link. There is no null field, no mode:checking conditional, and no confused state. New exam modes can be added as new classes without touching existing ones.
